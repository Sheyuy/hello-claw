## 1 From the "Server Room" to Your Pocket

### 1.1 The Agent's Journey to Slim Down

Recall the previous chapters:

- **OpenClaw**: Requires a Mac Mini ($599) or a cloud server
- **NanoClaw**: Can run on a standard server
- **ZeroClaw**: $10 hardware, <5MB memory
- **IronClaw**: Enterprise-grade security, requires PostgreSQL

They all pursue the same goal: **making Agents lighter, cheaper, and easier to deploy**.

But PicoClaw pushes this to the extreme — **it lets an Agent run on a "shrimp"**.

### 1.2 Why a "Shrimp"?

The name PicoClaw comes from "Pippi Shrimp," and the mascot is indeed a shrimp. The name is apt:

- **Shrimps are small**: PicoClaw uses <10MB of memory
- **Shrimps are cheap**: It can run on $10 hardware
- **Shrimps are flexible**: Supports multiple architectures (ARM, RISC-V, MIPS, x86)

### 1.3 What Does $10 Hardware Actually Mean?

What can $10 buy you?

- A Starbucks coffee
- A month of Netflix
- Or… a complete computer capable of running an AI Agent

PicoClaw can run on the **LicheeRV-Nano** — the E version costs $9.9 (with an Ethernet port) and the W version is $12.9 (with WiFi). It's smaller than a credit card, yet powerful enough to run Linux and PicoClaw.

Let's compare:

| Solution | Hardware Cost | Memory Usage | Power Draw | Use Case |
|----------|--------------|--------------|------------|----------|
| OpenClaw | $599 (Mac Mini) | >1GB | 10W+ | Desktop/Server |
| NanoClaw | $50 (Linux SBC) | ~100MB | 2-5W | Lightweight server |
| **PicoClaw** | **$10** | **<10MB** | **~3.5W** | **Embedded/IoT** |

**What does ~3.5W power draw actually mean?**

A 10,000mAh power bank (about $20) can supply power for **approximately 14 hours** (based on theoretical calculations at ~3.5W). If you want to build a portable AI assistant, this was simply not possible before.

> **Note**: Power consumption figures are theoretical estimates calculated from LicheeRV-Nano hardware specifications and PicoClaw's software resource usage. Actual battery life depends on specific workload and operating environment.

## 2 PicoClaw: An Agent Built for Hardware

### 2.1 Project Background

PicoClaw was developed by **Sipeed** — a well-known open-source hardware company focused on embedded development boards using RISC-V and ARM architectures.

Interestingly, **95% of PicoClaw's core code was generated by an AI Agent**. According to the project team, the developers instructed an AI to rewrite and optimize nanobot (a Python project from HKUDS) in Go, using its architecture as a reference. After "Human-in-the-loop" fine-tuning across multiple iterations, the current codebase took shape.

> **Note**: The claim that "95% of the code was AI-generated" is a statement made by the project team (Sipeed). See the official PicoClaw README for details.

This is itself a success story of agentic programming:
- **AI-generated**: 95% of the code was written by an Agent
- **Human review**: Critical logic and security checks were confirmed by humans
- **Continuous refinement**: Production-ready through multiple iterations

### 2.2 Extreme Performance Optimization

PicoClaw's numbers are remarkable:

| Metric | OpenClaw | nanobot (HKUDS) | **PicoClaw** |
|--------|----------|-----------------|--------------|
| **Language** | TypeScript | Python | **Go** |
| **RAM** | >1GB | >100MB | **<10MB** |
| **Startup** | ~30s | ~10s | **<1s** |
| **Cost** | $599 | ~$50 | **$10** |

> **Note**: Startup time and memory usage figures come from official documentation or community benchmark reports for each project. Actual performance depends on specific hardware configuration and workload.

**What does sub-1-second startup mean?** Even on a 0.6GHz single-core processor, PicoClaw completes its startup in under one second. This means it can wake up quickly, execute a task, and go back to sleep — perfect for battery-powered devices.

### 2.3 Architecture: Why Go?

PicoClaw chose Go for several key reasons:

**① Compiles to a single binary**

Go can cross-compile into statically linked binaries with no runtime dependency. This makes deployment trivially simple — just copy one file.

**② Low memory footprint**

Go's memory management is more efficient than Python or JavaScript. PicoClaw's core features require less than 10MB of memory, whereas an equivalent Python implementation might need 100MB or more.

**③ Cross-platform support**

Go natively supports cross-compilation to multiple architectures:
- x86_64 (Intel/AMD)
- ARM64 (Raspberry Pi, M1/M2 Mac)
- ARMv7 (older Raspberry Pi)
- RISC-V (LicheeRV, HiFive, etc.)
- MIPS (routers, embedded devices)

**④ Concurrency model**

Go's goroutines are lightweight threads that can support a large number of concurrent tasks with limited memory. This matters for an Agent that needs to handle multiple requests simultaneously.

## 3 Hardware Selection: Why Did ARM Win?

### 3.1 The Architecture Wars: x86 vs ARM vs RISC-V

In the embedded space, there are three dominant processor architectures:

**x86 (Intel/AMD)**
- ✅ High performance, mature ecosystem
- ❌ High power draw, expensive, generates significant heat
- Best for: Desktop computers, servers

**ARM**
- ✅ Low power draw, inexpensive, wide variety of options
- ✅ Mature ecosystem (Raspberry Pi, smartphones, IoT)
- Best for: Embedded systems, mobile devices, IoT

**RISC-V**
- ✅ Open-source, royalty-free, customizable
- ⚠️ Newer ecosystem, relatively less software support
- Best for: Education, research, purpose-built chips

### 3.2 Why ARM?

PicoClaw supports all three architectures, but **ARM is the most practical choice**. Here's why:

**① Mature ecosystem**

Over 50 million Raspberry Pis have been sold, and the software ecosystem around ARM Linux is extremely rich. Whatever functionality you can think of, there's almost certainly a ready-made solution.

**② Price-to-performance ratio**

You can get a capable ARM development board (Raspberry Pi Zero 2W, LicheeRV-Nano, etc.) for $10–50.

**③ Power efficiency**

ARM processors can achieve power draws below 1W, making them suitable for battery operation. x86 processors struggle to drop below 5W even at low load.

**④ Community support**

Solutions to problems are easy to find — simply because so many people use ARM.

## 4 Power Optimization: Running an Agent on a Battery

### 4.1 Why Does Power Draw Matter?

Consider these scenarios:

| Scenario | Description |
|----------|-------------|
| **Camping assistant** | You want an AI assistant in the wilderness, but there are no power outlets |
| **Smart home** | You want an Agent to control home devices without running power cables to every room |
| **Mobile robot** | You want to build a walking robot, but battery capacity is limited |

In these scenarios, **every milliwatt counts**.

### 4.2 PicoClaw's Power Efficiency Advantage

**By the numbers**:

| Device | Power Draw | Runtime on 10,000mAh Power Bank |
|--------|------------|----------------------------------|
| Mac Mini | 10–30W | 1–3 hours |
| Raspberry Pi 4 | 5–8W | 4–6 hours |
| Raspberry Pi Zero 2W | 1–2W | 15–20 hours |
| **LicheeRV-Nano + PicoClaw** | **~3.5W** | **~14 hours** |

**How is ~3.5W achieved?**

1. **Go's efficiency**: No virtual machine overhead, relatively low garbage collection overhead
2. **Stripped-down features**: Only core functionality is retained; unnecessary dependencies are removed
3. **Event-driven design**: Sleeps when idle, wakes up quickly when an event occurs
4. **Lightweight models**: Supports local small models (e.g., TinyLlama), reducing network requests

### 4.3 In Practice: Tips for Extending Battery Life

If you want to run PicoClaw on a battery, consider these optimizations:

**① Intermittent operation**

There's no need to run 24/7. Set a scheduled wake-up (e.g., check every 15 minutes) and sleep the rest of the time.

**② Lower the CPU frequency**

Most development boards allow you to reduce the CPU frequency. For non-demanding tasks, 0.6GHz is sufficient.

**③ Disable unnecessary peripherals**

- Not using HDMI? Disable GPU output.
- Not using USB? Disable the USB controller.
- Not using WiFi? Use a wired connection or connect intermittently.

**④ Use low-power CPU scaling modes**

Linux offers several power-saving governor modes:
- `powersave`: Always runs at minimum frequency
- `ondemand`: Ramps up frequency as needed
- `conservative`: Raises frequency gradually

For Agent workloads, `conservative` is usually the best choice.

## 5 Offline Capability: An Agent Without Internet

### 5.1 Why Go Offline?

You might ask: don't all devices have internet connectivity nowadays? Why bother going offline?

| Scenario | Description |
|----------|-------------|
| **Privacy protection** | You don't want personal data sent to the cloud. Running locally means your conversations, files, and habits stay on your device |
| **Unreliable connectivity** | You're in a remote area, a basement, or somewhere with a poor signal |
| **Cost considerations** | Every API call costs money. Handling simple tasks locally can dramatically reduce costs |
| **Response speed** | Local models respond faster — no network round-trip needed |

### 5.2 Local Model Options

PicoClaw supports several local model deployment approaches:

**① Ollama**

```bash
# Install Ollama on the device running PicoClaw
ollama run tinyllama

# Then point PicoClaw's config to the local Ollama instance
```

**② llama.cpp**

An inference engine optimized for embedded devices, with support for quantized models (4-bit, 5-bit). A quantized 7B model requires approximately 4GB of memory; smaller models under 1B (e.g., TinyLlama) can run on devices with 1–2GB of memory (2-bit quantization can run on 512MB devices, though with performance limitations).

**③ vLLM**

If you have sufficient memory (2GB+), vLLM delivers faster inference speeds.

### 5.3 The Capability Boundaries of an Offline Agent

Local models are powerful, but they have limitations:

**✅ What local models are good at:**
- Text generation and summarization
- Simple question answering
- Code completion
- Format conversion

**❌ What local models struggle with:**
- Tasks requiring up-to-date knowledge (e.g., "today's news")
- Complex reasoning
- Multilingual translation (small models produce lower quality results)

**Hybrid approach:**

PicoClaw supports a "local-first" strategy:
- Simple tasks → handled by the local model
- Complex tasks → call a cloud API
- No network available → run fully locally (with degraded functionality)

## 6 Summary: The "Democratization" of Agents

PicoClaw represents a new direction for Agent technology: **extreme lightweight design and low cost**.

It proves several things:

1. **Agents don't need expensive hardware** — a $10 device is sufficient to run core functionality
2. **Low power draw is achievable** — sub-1W power consumption makes battery operation a reality
3. **Offline operation is viable** — local models can satisfy basic needs

What does this mean for the industry at large?

**For developers**: You can embed Agents into scenarios that were previously impossible — smart home devices, wearables, robots…

**For users**: AI assistants are no longer just "cloud services" — they are software you can truly **own**. It runs on your device, requires no subscription fee, needs no internet connection, and keeps your data completely private.

**For society**: The democratization of AI technology. Running AI is no longer the exclusive domain of large corporations; ordinary people can experience this technology on a $10 device.
